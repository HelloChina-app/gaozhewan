## n8n 是什么，为什么总和 Make 比

n8n 是面向技术团队的工作流自动化平台。它同样用画布连接触发器和动作，但更强调数据表达式、JavaScript 或 Python 代码、任意 HTTP 请求、自定义节点与自托管。你可以购买 n8n Cloud，也可以运行标准自托管 Community Edition；两条路线拥有不同的费用、协作功能和运维责任。

人们常把 n8n 称为 Make 的“开源替代”，这个说法过于简单。更准确的差别是：n8n 给技术团队更多部署和代码控制，Make 把更多基础设施与应用连接体验交给 SaaS。自托管 n8n 不等于零成本，也不自动拥有商业计划中的所有协作、治理和扩展功能。

## Workflow、Node、Item 和 Execution

Workflow 是完整自动化；Node 是触发、读取、转换或写入的一步；Item 是节点之间流动的数据对象；Execution 是工作流从开始到结束的一次运行。表达式可以读取前序节点数据，Code 节点能执行自定义逻辑，HTTP Request 则在没有现成集成时直接调用 API。

n8n 当前付费计划以每月 workflow executions 为主要计费口径，而不是对工作流里的每一个步骤分别收费。官方定价页说明，一次完整运行无论有多少步骤，通常算一次 execution；套餐、并发、历史保留、项目和企业能力仍有差异，购买前必须查看实时定价。

::: callout 自托管不是“无限免费 SaaS”
Community Edition 让你运行自己的标准实例，但服务器、数据库、域名、TLS、邮件、备份、升级、安全响应和故障恢复都由你承担。高级协作、SSO、环境、Git 版本控制和扩展能力可能属于商业计划。
:::

## 第一个 n8n 工作流怎样设计

从一个可恢复任务开始，例如“Webhook 接收线索，校验后写入 CRM，并发送内部通知”。先在纸上定义输入结构、唯一 ID、成功输出和三类失败，再创建工作流。

- Webhook Trigger 接收测试事件，并限制调用来源或增加认证。
- Edit Fields 或 Code 节点把外部字段转换成内部统一结构。
- IF 节点拒绝缺少关键字段的 Item，把原因送到错误分支。
- CRM 或 HTTP Request 节点写入数据，使用外部事件 ID 防止重复创建。
- 通知节点只发送结果和可追踪 ID，不复制不必要的个人信息。
- 单独配置 Error Workflow，记录失败节点、execution ID 和恢复入口。

测试时不要只点击一次成功样本。至少覆盖重复事件、目标 API 限流、凭据失效、网络超时和部分数据为空。生产环境还要决定失败后自动重试还是人工重放；无幂等保护的自动重试可能重复发邮件、创建订单或扣减库存。

## n8n 的 AI Agent 能做什么

n8n 的 Advanced AI 能把聊天模型、记忆、检索器和工具节点连接成工作流。AI Agent 可以根据输入选择工具，普通应用节点也可作为 Agent 的可调用能力；平台文档还提供人工审核工具调用的流程，让 Gmail 等高影响动作在发送前暂停等待批准。

适合交给 Agent 的任务包括分类、抽取、检索和选择下一步工具。付款、删除、公开发布、批量外联和权限修改不应只依赖模型自行判断。把“生成建议”和“执行外部动作”拆成两个阶段，限制工具参数范围，并保留输入、模型输出、实际调用和批准人的审计记录。

## 自托管生产环境必须补什么

最小可用的生产部署至少需要稳定数据库、持久卷、HTTPS、固定加密密钥、定期备份和升级流程。不要只备份 workflow JSON；凭据加密依赖、数据库、二进制数据和环境配置也决定能否恢复。恢复演练应在另一台实例上完成，而不是只确认备份文件存在。

关闭公开注册，限制编辑器与 Webhook 的网络入口，对反向代理设置请求大小和超时；给 OAuth 与 API Key 使用最小权限，并按环境隔离。官方提供 `n8n audit`、API 和节点方式的安全审计，可检查未使用凭据、危险节点、社区节点、文件系统访问、未保护 Webhook 和缺失设置。审计报告是线索，不替代补丁、日志与人工评审。

## 数据、日志与执行历史

自动化经常处理邮箱、客户、订单和内部文档。先确定哪些字段必须进入 n8n，哪些可以只传引用 ID。生产日志不应长期保存完整敏感 payload；执行历史保留时间应与排错、合规和存储成本一起决定。

包含图片、音频或大文件时，要特别关注二进制数据存储与清理。部分外部对象存储和规模化能力属于特定商业层级，不能看到“自托管”就假设所有企业能力都在 Community Edition。升级前阅读对应版本说明，并保证主实例、worker 和 runner 的版本一致。

## n8n 与 Make 怎么选

选择 n8n，如果团队能维护服务、需要自托管或深度代码处理、工作流步骤多而执行次数相对可预测，或需要把自动化纳入开发与安全流程。选择 Make，如果希望几乎不运维、业务人员直接维护画布、主要使用成熟 SaaS 集成，并愿意接受 credits 与 bundle 带来的用量模型。

同一个流程应同时估算两类成本。Make 要计算每个模块处理的 bundle 和动态 AI credits；n8n Cloud 要计算完整 execution、并发与套餐能力；n8n 自托管还要加入服务器、数据库、备份、监控、升级和工程师时间。对小团队来说，月费更低但每月多花半天维护，往往不是更便宜。

## 什么时候不该上 n8n

只有一个简单的“收到表单发通知”需求、团队无人维护服务器、数据没有稳定唯一键，或业务无法接受自行处理安全更新时，不要因为热门教程直接自托管。先用托管服务验证流程价值，等执行量、数据控制或扩展需求真正出现，再迁移基础设施。

一个可靠的 n8n 工作流应该具备明确输入、幂等写入、错误路线、最小权限、可观测 execution 和恢复手册。画布上所有节点变绿只是开发阶段完成；能在凭据失效、API 限流和实例重启后安全恢复，才算进入生产。
